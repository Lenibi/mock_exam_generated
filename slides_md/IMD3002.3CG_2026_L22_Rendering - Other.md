## Slide 17

Non-Photorealistic Rendering
Silhouette Edge Rendering
There are a number of different types of edges that can 
be used in toon rendering:
◼A boundary  or border edge  ~ one that is not shared by two 
polygons (e.g. a paper edge), a solid object typically has no 
boundary edges
◼A crease  or a hard edge  ~ one that is shared by two 
polygons, but the angle between them is greater than some 
defined value (e.g. 60 °) – alternatively we can look at the 
vertex normals  (e.g. a cube has crease edges)

---

## Slide 18

Non-Photorealistic Rendering
Silhouette Edge Rendering
Toon Edge types (Cont’d):
◼A material edge  ~ which appears when two polygons sharing 
an edge differ in material – this can also be an edge that an 
artist wants to display (e.g. forehead lines or a line to 
separate the same coloured pants and shirt)
◼A silhouette edge  ~ in which two neighbouring polygons face 
in different directions compared to some direction vector
This categorisation is based on common usage, but there 
are some variations (and mixing)

edges the other techniques miss

This group of machines is called a “ Farm ”

## Slide 52

Network Rendering
Introduction
Network Rendering is generally started by a workstation 
on a client, which is where the scene has been created (or

However, we often use another machine to actually

## Slide 53

Network Rendering
Introduction
It is usual for the “ Master ” to also perform rendering and 
have a “ slave ” running at the same time – although this 
depends on how big the farm is: the more nodes in the 
farm, the more work the Master has to do just making 
sure everything is working
The “Master” has two jobs, the first (and foremost) is to 
distribute all relevant information to the “Slaves” in the 
farm
The second is to resubmit a job if a slave fails to render 
the frame

---

## Slide 54

Network Rendering
Introduction
Failure is rare, but not totally absent, for large farms 
(small ones too); this can be for several reasons:
◼Transmission Error ~ each node relies heavily on the data it is 
presented, it has no verification method, therefore if there is an 
error in transmission (i.e. a read from disk, etc) then an error on 
that node can occur
◼Computational Error ~ GI is complex and involves many 
calculations; it is rare, but it can exist that the calculations for a 
particular frame accumulate enough errors for the frame, or 
the node, to crash

## Slide 65

Network Rendering
Complexity
There are several factors that influence the render time; 
whilst there are no hard rules, it is good to remember the 
following:
◼Number of Polygons  (tessellated or otherwise)
◼Number of Lights  (each light means more rays)
◼Screen Size  (each pixel in the viewport means another ray to 
trace)
◼Map Sizes  (the larger the number of pixels in a texture, the 
more time it takes to calculate)
◼CPU/Memory Capacity  (faster CPU is obvious, but if we run 
out of memory, the system will start swapping to disk, which is 
terribly slow …)

DR works on the basis of each frame from a scene being 
broken down into sections (called Buckets) and each 
bucket being sent (via the network) to a different machine 
to be rendered
When that frame has been completed, the machines start 
on the next frame

## Slide 72

Distributed Rendering
Introduction
This is very different to Network Rendering because we 
are getting all machines to work on one frame at the 
same time
This changes the considerations for network and storage, 
because we have a massive “pulse” of data all at once – 
in this respect this limits the number of machines that can 
work on a frame at one time (although sometimes this is a 
physical limit, not a software limit)

## Slide 74

Distributed Rendering
Disadvantages
The disadvantages of DR are as follows:
◼Limited number of machines ~ as we are rendering a frame 
on many machines, we can in fact have a higher data rate 
(easily approach 373MBytes/second of real -time)
◼Even though it’s a smaller part of the frame, the entire scene is still 
send to each machine
◼Larger overhead  ~ in NR the overhead is per frame, so we 
spend time to sort the frames and distribute them to each 
slave; in DR we separate this into buckets and therefore per 
frame we have an overhead accumulated per bucket
◼This is from the network protocol and the render farm communication 
requirements in general

## Slide 75

Distributed Rendering
Disadvantages
Disadvantages cont’d
◼Bigger Error Margin  ~ if a frame fails in NR, it can be simply 
re-rendered; however, if a bucket fails there is either an error 
in the frame, or it has to be re -rendered (which can result in a 
higher overhead)

## Slide 76

Distributed Rendering
Advantages
DR is fairly new, but it is useful (otherwise it wouldn’t be 
used)
◼Inter-Frame Dependency  ~ As we are rendering 1 frame at a 
time, we can actually avoid the problems of inter -frame 
dependency because we are only rendering 1 frame at a time
◼Faster Frames  ~ sometimes we can only see when something 
has gone wrong once we have rendered a frame in GI – and 
those frames can take a long time; using DR we can see that 
frame much quicker and therefore reduce bugs quicker
